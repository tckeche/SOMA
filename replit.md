## Overview
SOMA is a full-stack intelligent assessment platform designed for school-level mathematics and science assessments. It features interactive MCQ quizzes with LaTeX rendering for students, an AI-powered copilot for tutors to build curriculum-aligned quizzes, and super-admin tools for user and content management. The platform is built upon a structured curriculum catalogue to ensure all AI-generated content is syllabus-aligned, auditable, and accurate. Its core purpose is to provide correct and verifiable assessments by grounding every AI interaction in specific syllabus contexts.

## System Architecture

### UI/UX
The frontend is built with React, Vite, Tailwind CSS, and Shadcn UI. Mathematical content is rendered using ReactMarkdown, remark-math, and rehype-katex. Graphs are handled by the `GraphPlot` component. The design features glassmorphism on authentication and tutor surfaces, with a plaque-first tutor dashboard incorporating flip animations and subject-specific styling.

### Backend
The backend uses Node.js and Express, with Drizzle ORM managing a PostgreSQL database (hosted on Supabase). Authentication is handled via Supabase Auth, integrated with a `soma_users` table that supports `student`, `tutor`, and `super_admin` RBAC roles. Frontend routing is managed by wouter.

### Curriculum Catalogue
A normalized curriculum catalogue (`examining_bodies` â†’ `levels` â†’ `subjects` â†’ `syllabi` â†’ `topics` â†’ `subtopics` â†’ `learning_requirements` â†’ `competencies` â†’ `papers`) serves as the canonical source of truth for subject and syllabus context. This catalogue directly informs the AI pipeline through a `CatalogueCopilotContext` payload, which generates a deterministic prompt block for LLM user messages.

### Reference Text & Semantic Retrieval
The system deterministically builds and embeds reference text "chunks" per (topic, tier) pair using OpenAI's `text-embedding-3-small`. Semantic search filters these chunks by metadata and uses cosine similarity with a lexical keyword boost. This allows for auto-population of `selectedTopics` in AI prompts based on tutor queries, providing explicit context to the LLM. A `stripSyllabusNoise` pre-processor cleans raw syllabus documents before AI processing.

### AI Generation Pipeline
The AI pipeline operates in a two-stage Maker â†’ Verifier model, enforcing a strict no-self-grading rule:
1.  **Maker**: Claude Sonnet 4.6 (with GPT-4o as fallback) drafts MCQs without explanations.
2.  **Verifier**: GPT-4o (with Gemini 2.5 Flash as fallback) verifies quizzes generated by Claude, fixes errors, and writes final explanations. Gemini 2.5 Flash verifies GPT-4o generated quizzes.
Deterministic guards (`applyDeterministicIntegrityGuards`, `validateAndCorrectMcqAnswers`, `applyMathValidatorCorrections`) ensure data integrity by deduplicating options, clamping marks, and correcting answers. Quizzes exceeding 15 questions are batched. Progress and warnings are streamed via Server-Sent Events to the builder Co-Pilot.

### Syllabus Insights
The `syllabusInsights.ts` service generates a per-student topic-coverage radar and a paper-readiness heatmap by analyzing student reports against the Cambridge syllabus catalogue. These insights are displayed on student and tutor dashboards.

### Tutor Builder & Copilot
The Tutor Portal provides a Command Centre dashboard. The AI Copilot (Builder) is a catalogue-driven wizard that assists tutors in creating quizzes, saving typed drafts to an in-memory layer until explicit publication. AI Dashboard Intelligence synthesizes assessment data into narratives, weakness lists, and at-risk flags.

### Student Experience
The Soma Quiz Engine offers a LaTeX-aware MCQ player with a summary view. The Soma Quiz Review provides post-quiz analysis, including explanations, correct/incorrect markers, and AI feedback.

### Authentication System
A single `/login` route handles tab-based login, signup, and forgot-password functionalities, with inline error handling.

## Secrets handling
All credentials â€” database URLs, API keys, JWT/session secrets, admin
passwords â€” live **only** in the Replit Secrets store. They must never
appear in `.replit`, `.env`, `docker-compose.yml`, scripts, fixtures,
test mocks, comments, or any other in-repo file. Use the env-secrets
request flow when a new credential is needed at runtime; if a plaintext
credential is found in the tree, treat it as compromised: rotate
upstream first, then remove the in-repo copy, then scrub git history.

## Database migrations policy
Schema changes use **hand-crafted, additive SQL migrations** in
`migrations/NNNN_*.sql` (e.g. `0006_pg_trgm.sql`). The `db:push --force`
shortcut is **not used** â€” the cumulative DDL is large enough that
auto-generated diffs would be destructive. New migrations should be
idempotent (`CREATE EXTENSION IF NOT EXISTS â€¦`, `ADD COLUMN IF NOT
EXISTS â€¦`) so they can re-run safely.
